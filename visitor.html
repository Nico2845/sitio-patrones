Patrón de comportamiento Lámina 01

Visitor

Cómo agregar nuevas operaciones sobre distintos tipos de documento sin llenar el generador de reportes de condicionales.

1 El problema

Un ReportGenerator recibe documentos de distinto tipo — facturas (Invoice), recibos (Receipt) — y necesita generar un reporte distinto según el tipo. La solución rápida es un bloque de if ($document instanceof Invoice) … if ($document instanceof Receipt) … dentro del propio generador.

El problema: eso viola el principio Open/Closed de SOLID. Cada vez que aparece un nuevo tipo de documento, hay que volver a abrir y modificar ReportGenerator para agregarle un nuevo if — y el riesgo de romper la lógica de los tipos ya existentes crece con cada cambio.

2 La solución

Cada tipo de documento implementa accept(visitor), y delega en el visitante la lógica de cómo generarse. El DocumentVisitor declara un método por cada tipo (visitInvoice, visitReceipt), y ReportGenerator pasa a ser un visitante concreto que implementa esa interfaz. Ya no hay if (instanceof …): el propio documento sabe a qué método del visitante llamar.

3 Código mínimo

after.php solución con Visitor
// Cada tipo de documento pertenece a una interfaz que acepta un visitor
interface Document
{
    public function accept(DocumentVisitor $visitor);
}

class Invoice implements Document
{
    public function accept(DocumentVisitor $visitor)
    {
        return $visitor->visitInvoice($this);
    }
}

class Receipt implements Document
{
    public function accept(DocumentVisitor $visitor)
    {
        return $visitor->visitReceipt($this);
    }
}

interface DocumentVisitor
{
    public function visitInvoice(Invoice $invoice);
    public function visitReceipt(Receipt $receipt);
}

class ReportGenerator implements DocumentVisitor
{
    public function visitInvoice(Invoice $invoice)
    {
        return "Generating invoice report";
    }
    public function visitReceipt(Receipt $receipt)
    {
        return "Generating receipt report";
    }
}

$generator = new ReportGenerator();
echo (new Invoice())->accept($generator);

4 Cuándo sí / cuándo no

SÍ usarlo cuando…

  • Van a existir varios tipos de documento/objeto y esa lista puede crecer.
  • Se necesita agregar operaciones nuevas sin modificar las clases existentes.
  • Las operaciones son independientes entre sí (reportes, exportar, validar…).

NO usarlo cuando…

  • Solo hay un tipo de documento y no se espera que crezca.
  • Introducir Visitor solo agrega clases e interfaces sin beneficio real (YAGNI).
  • Los tipos de objeto cambian más seguido que las operaciones — sería al revés.
before.php code smell
class ReportGenerator
{
    public function generate($document)
    {
        // viola OCP: hay que tocar este método
        // cada vez que aparece un nuevo tipo
        if ($document instanceof Invoice) {
            return "Generando reporte de invoice";
        }
        if ($document instanceof Receipt) {
            return "Generando reporte de receipt";
        }
        throw new Exception("Tipo inválido");
    }
}
no_refactor.php cuándo NO conviene
// Un solo tipo de documento: Visitor
// solo agregaría complejidad sin beneficio
class Invoice
{
    public function generateReport()
    {
        return "Generating invoice report";
    }
}

$invoice = new Invoice();
echo $invoice->generateReport();

5 Un ejemplo real

Laravel no trae Visitor de forma nativa, pero existe el paquete LaravelVisitor, que permite aplicarlo para procesar colecciones de elementos sin condicionales repetidos.

Java tiene un ejemplo "oficial" en la interfaz javax.lang.model.element.ElementVisitor, usada para recorrer elementos del código (clases, métodos, interfaces) durante el procesamiento de anotaciones, ejecutando distintas operaciones según el tipo exacto de un elemento que no se conoce en tiempo de compilación.
