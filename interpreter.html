Patrón de comportamiento Lámina 02

Interpreter

Cómo evaluar expresiones aritméticas sin un switch que se vuelve más largo cada vez que aparece un operador nuevo.

1 El problema

Un ExpressionEvaluator recibe un operador ("+", "-") y dos números, y usa un switch para decidir qué operación aplicar. Funciona — hasta que alguien pide un operador que el switch no contempla (ej. multiplicación, o algo como "greater") y el código revienta con una excepción.

El dolor real: cada operador nuevo obliga a volver a abrir ExpressionEvaluator y agregarle un case más. El switch crece indefinidamente y toda la lógica de evaluación vive amontonada en un solo lugar.

2 La solución

Cada operación se vuelve su propia clase que implementa una interfaz común Expression con un método interpret(). Los números son expresiones "hoja" (Number), y las operaciones (AdditionExpression, SubtractionExpression, MultiplicationExpression) son expresiones "compuestas": guardan dos expresiones (left, right) y se interpretan llamando recursivamente a interpret() de cada una. Agregar un operador nuevo es agregar una clase — no tocar un switch existente.

3 Código mínimo

after.php solución con Interpreter
// Cada expresión ahora es su propia clase
interface Expression {
    public function interpret();
}

class Number implements Expression
{
    public function __construct(private int $value) {}

    public function interpret(): int
    {
        return $this->value;
    }
}

// Usa recursividad para interpretar una cadena de expresiones
class AdditionExpression implements Expression
{
    public function __construct(
        private Expression $left,
        private Expression $right
    ) {}

    public function interpret(): int
    {
        return $this->left->interpret() + $this->right->interpret();
    }
}

// SubtractionExpression y MultiplicationExpression siguen la misma forma
// (mismo constructor, cambia solo el operador en interpret())

$result = new AdditionExpression(
    new Number(10),
    new Number(5)
);

echo $result->interpret(); // 15

4 Cuándo sí / cuándo no

SÍ usarlo cuando…

  • Hay que evaluar reglas o expresiones que cambian/crecen con frecuencia.
  • La gramática (conjunto de operaciones) es pequeña y bien definida.
  • Se quieren componer expresiones anidadas de forma recursiva.

NO usarlo cuando…

  • Solo hay una operación (ej. sumar) y no se espera que crezca.
  • La gramática crecería demasiado — las clases se multiplican sin control.
  • Introducirlo solo complicaría el código sin beneficio real (YAGNI).
before.php code smell
class ExpressionEvaluator
{
    public function evaluate(string $operator, int $left, int $right): int
    {
        // switch que crecerá en el futuro
        switch ($operator) {
            case '+':
                return $left + $right;
            case '-':
                return $left - $right;
            default:
                throw new Exception("Unknown operator");
        }
    }
}
no_refactor.php cuándo NO conviene
// Solo se hace una suma; no hay un set de
// expresiones que crecerá. Interpreter aquí
// solo complicaría el código sin beneficio
function add(int $left, int $right): int
{
    return $left + $right;
}

echo add(10, 5);

5 Un ejemplo real

Un gran ejemplo en el mundo real lo encontramos en el framework Symfony, una de las herramientas de desarrollo web más importantes de la actualidad. Cuenta con un componente llamado ExpressionLanguage.

Este motor permite a las empresas escribir reglas de negocio complejas en texto (por ejemplo, para calcular descuentos dinámicos en una tienda online) y, por detrás, usa el patrón Interpreter para convertir ese texto en un árbol de objetos que se evalúa de forma segura, limpia y ultra mantenible.
